# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Overview

`devices/gl` is the OpenGL ANARI device (`libanari_library_visgl`). It implements the ANARI C API as a rasterization renderer targeting OpenGL 4.3 or GLES 3.2. See the parent repo's CLAUDE.md for build instructions; this device is enabled by `-DVISRTX_BUILD_GL_DEVICE=ON` (default ON).

## Code Generation

A significant portion of the source is auto-generated by the ANARI SDK's CMake helpers:
- `anari_generate_frontend()` → `generated/VisGLObjects.h/cpp`, `VisGLParameter.h`, `VisGLString.h/cpp`, `VisGLDeviceFactories.cpp`
- Input schemas: `visgl_*.json` files in the device root

**Do not hand-edit files in `generated/`.** Modify the corresponding `.json` schema files and rebuild instead.

## Architecture

### Object Lifecycle

All ANARI objects inherit from `ObjectBase` (`src/VisGLObject.h`). The typical pattern:

```cpp
template<class T, class B = ObjectBase>
class DefaultObject : public B {
    T staging;      // params set via setParameter(), not yet visible
    T current;      // params after commitParameters()
};
```

`setParameter()` writes to `staging`; `commitParameters()` copies `staging → current`. Objects track changes via `lastEpoch`.

Object storage uses `ConcurrentArray<ObjectBase>` (`src/ConcurrentArray.h`) — a lock-free, exponentially-growing pool (max 64 blocks). ANARI handles are indices into this array. Reference counting splits internal refs (upper 32 bits) from external/app refs (lower 32 bits).

### Rendering Pipeline

Rendering is driven by `VisGLFrameObject` → `VisGLWorldObject`. Each `renderFrame()`:
1. **Scene collection** — `CollectScene` visitor walks the `World → Group → Instance → Surface` hierarchy
2. **Shadow pass** — depth render from directional light viewpoint (if `shadowMapSize > 0`)
3. **Main render** — per-surface: bind geometry VAO, compile/bind material shader, draw
4. **Occlusion bake** — if `occlusionMode` ≠ `"none"`, bakes per-vertex/primitive AO into geometry

### Shader System

Shaders are assembled from segments (`src/AppendableShader.h`, up to 63 segments). Geometry contributes vertex/fragment code; materials contribute fragment declarations and body. Compiled programs are cached by their segment combination hash. Resource binding slots (SSBO indices, texture units, attribute locations) are constants in `src/shader_blocks.h`.

`DrawCommand` (`src/DrawCommand.h`) bundles all GPU state for a draw call: shader, VAO, up to 10 textures/samplers, up to 10 SSBOs, uniforms, primitive type, and instance count.

### GL Context Management

`VisGLDeviceObject` (`src/VisGLDeviceObject.h/cpp`) manages context creation with this priority:
1. GLX context current on calling thread → create sibling context on internal render thread
2. EGL display current → use it
3. No display → create offscreen EGL context
4. `"EGLDisplay"` device parameter → explicit override

Backends: `src/egl/`, `src/glx/`, `src/wgl/` (Windows stub only). The abstract interface is `src/glContextInterface.h`. All GL calls happen on the device's internal thread via `queue_thread` (`src/queue_thread.h`).

### Key Device Parameters

Set via `anariSetParameter(device, device, ...)`:
- `"glAPI"` (string): `"OpenGL"` or `"OpenGL_ES"`
- `"glDebug"` (bool): forward GL debug output to ANARI status callback
- `"EGLDisplay"` (pointer): explicit EGL display

Renderer parameters (`renderer` object, type `"default"`):
- `"shadowMapSize"` (int32): shadow map resolution (0 = implementation-defined)
- `"occlusionMode"` (string): `"none"` | `"incremental"` | `"firstFrame"`

## Known Limitations

WGL/Windows not implemented. The following reported extensions are aspirational (may be incomplete): cone/curve/quad geometry, omnidirectional camera, indexed spheres. Only directional lights cast shadows.
